Andrew shared a full analysis of a proposed "Recon" feature for DealerScope.
This was previously discussed with Claude/ChatGPT in a separate session.

### Summary
Recon = manual vehicle evaluation workbench / analyst bench.
Bridges the gap between raw scraper discovery and actual buying decision.
Lets you evaluate ANY vehicle (GovDeals, Copart, IAA, Craigslist, Facebook, etc.) — not just scraped ones.

### Verdict
SHIP as "Recon Beta / Manual Evaluator" — NOT as the main scoring authority.

### What Recon Does
- Manual input: VIN, make/model/year/mileage/condition/asking price/location
- VIN decode autofills fields
- Runs through pricing ladder: retail comps → Manheim → internal model
- DOS scoring + bid ceiling + profit range (pessimistic/optimistic)
- Verdict: BUY / WATCH / PASS
- Writes to recon_history table
- Optional "promote to pipeline" → opportunities table
- Telegram/OpenClaw workflow integration

### Red Team Issues (to fix before shipping)
1. Over-anchors on fixed DOS weights + fixed 0.88 bid ceiling — needs confidence-tiered logic
2. Falls back to internal_model too easily — dangerous if users trust it as truth
3. Transport costing uses state centroids not real routing
4. Trim/VIN underused — King Ranch vs XL pricing gap not captured
5. Condition notes collected but not converted to structured penalties
6. Dev-mode auth fallback is a production risk
7. UX implies more precision than model deserves — crisp BUY/PASS without confidence caveat

### Recommended Additions
- Valuation reliability class: A (retail comps + VIN) → B (Manheim) → C (model est) → D (sparse)
- Block one-click promotion for C/D without user confirmation
- Confidence badge more prominent than verdict
- Self-calibration: track predicted vs realized profit/days-to-sell
- Rate limiting + audit logs for evaluate/promote

### New Tables Needed
- recon_history (all evaluations)
- recon_history → opportunities (promotion path)
